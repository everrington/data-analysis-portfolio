Product Price Sensitivity
by Promotion Type

BigQuery SQL Price Elasticity Promotion Analytics Product Intelligence

Promotions reduce price — but not every product responds the same way. A deep discount on a commodity SKU drives volume; the same discount on a premium SKU might simply erode margin without moving units. Without a systematic way to measure this, commercial teams are left guessing which promotions actually work and which just give revenue away.

This query builds a product-level price sensitivity view that compares each SKU's average daily sales and ASP during promotional periods against its own non-promotional baseline — across every promotion type it has participated in over the past 364 days. The result is a per-SKU, per-promotion-type elasticity score that quantifies how sales respond to price changes, enabling data-driven decisions on which promotions to run, extend, or retire.

The pipeline carefully handles two tricky edge cases: SKUs that have never had a normal period (always-on promotions / EDLP items) are excluded from the normal vs. promo comparison; and overlapping promotions on the same day are tracked and flagged so the baseline ASP is always the minimum active promotion price — preventing double-counting.

SQL walkthrough

The query is structured as a chain of CTEs. Each section below explains the logic before showing the code.

01 Variables, product, branch & member filters

Define the 364-day rolling window anchored to today. Product master is filtered to active, sellable units only — inactive statuses like cancelled, discontinued, or clearance are excluded. Branch master excludes specific store formats. First sale date per material code is captured here to later trim the date spine to each SKU's actual commercial life.

BigQuery SQL
WITH _var AS (
  SELECT
    DATE_SUB(CURRENT_DATE("Asia/Bangkok"), INTERVAL 364 DAY) AS std,
    CURRENT_DATE("Asia/Bangkok")                                  AS endd
),

-- Active, sellable products only
-- Capture first sale date per material for spine trimming later
_product AS (
  SELECT DISTINCT
    *,
    MIN(first_sales_date) OVER (PARTITION BY material_code) AS fsd
  FROM `[project].[dataset].product_master`
  WHERE
    unit_code = "ST"
    AND product_status NOT IN (
      "C","CD","CP","CP2","PM","TD","OLC","D","PMT","CPT","CDT"
    )
),

_branch AS (
  SELECT *
  FROM `[project].[dataset].branch_master`
  WHERE storetype NOT IN ("[Excluded Format]")
),
02 Promotion flattening to daily grain

Promotions are stored as date ranges (start → end). This step unnests each promotion into individual daily rows using GENERATE_DATE_ARRAY, so every day a product is on promotion becomes a single joinable record. Three important things happen here: promotion sub-themes are normalized into clean labels (e.g. anything ending in "MAIL" → "MAIL"); overlapping promotions on the same barcode-day are counted; and a QUALIFY filter keeps only the minimum price when multiple promotions apply on the same day — preventing artificially high ASP on promo days.

BigQuery SQL
_pro AS (
  SELECT
    promotion_id,
    barcode,
    doc_date,
    promotion_price,
    promotion_theme,
    -- How many distinct promo themes are active for this barcode on this day?
    COUNT(DISTINCT promotion_theme)
      OVER (PARTITION BY doc_date, barcode)              AS overlap,
    -- Count of short-cycle themes (used for campaign classification)
    COUNT(DISTINCT
      CASE WHEN promotion_theme IN (
        "[Theme A]","[Theme B]","[Theme C]","7 DAY","[Theme D]"
      ) THEN promotion_theme END)
      OVER (PARTITION BY doc_date, barcode)              AS short_pro

  FROM (
    SELECT DISTINCT
      promotion_id,
      promotion_sub_theme,
      barcode,
      start_date,
      end_date,
      promotion_price,
      GENERATE_DATE_ARRAY(start_date, end_date)          AS doc_date,
      -- Normalize sub-theme labels into clean promotion categories
      CASE
        WHEN promotion_sub_theme LIKE "%MAIL"      THEN "MAIL"
        WHEN promotion_sub_theme LIKE "%7 DAY"     THEN "7 DAY"
        WHEN promotion_main_theme LIKE "Others"    THEN "Others"
        ELSE promotion_sub_theme
      END AS promotion_theme

    FROM `[project].[dataset].promotion_master_daily`
    WHERE
      all_store_flag = "All Store"
      AND promotion_sub_theme NOT IN (
        "[Excluded Theme A]",  -- bias from low stock / exclusive price
        "[Excluded Theme B]",  -- store format with incomplete data
        "[Excluded Theme C]"   -- statistical outlier
      )
      AND barcode IS NOT NULL

  ), UNNEST(doc_date) doc_date

  GROUP BY ALL

  -- Keep only the minimum promotion price when multiple promos overlap on same day
  QUALIFY promotion_price = MIN(promotion_price) OVER (PARTITION BY barcode, doc_date)
),
03 Transaction aggregation to daily sales + ASP

Aggregate transactions to barcode × date level, computing total sales (excl. VAT) and average selling price. The join to product and branch master ensures only active products in eligible store formats are included. No customer join is needed here — this is a product-level, not a customer-level, analysis.

BigQuery SQL
_receipt AS (
  SELECT DISTINCT
    r.barcode,
    r.doc_date,
    SUM(sales_exc_vat)                                   AS sales,
    SUM(sales_exc_vat + sales_vat) / SUM(sales_qty)      AS asp

  FROM `[project].[dataset].transactions` r
  INNER JOIN _product p ON r.barcode      = p.barcode
  INNER JOIN _branch  b ON r.branch_code  = b.branch_code

  WHERE
    r.doc_date BETWEEN (SELECT std  FROM _var) AND (SELECT endd FROM _var)
    AND sales_qty     <> 0
    AND sales_exc_vat <> 0

  GROUP BY ALL
),
04 Full SKU × date spine

A cross join between a pre-built date array and the product master generates a complete skeleton of every barcode × date combination in the window. This is critical — without it, days with zero sales would be invisible in the transaction table, causing normal-period averages to be inflated. The date spine ensures every day is accounted for, including zero-sales days.

BigQuery SQL
_id AS (
  SELECT DISTINCT
    barcode,
    d.doc_date
  FROM `[project].[dataset].date_array` d
  -- Cross join: every product gets every date in the window
  CROSS JOIN _product p
  WHERE
    d.doc_date BETWEEN (SELECT std FROM _var) AND (SELECT endd FROM _var)
),
05 Period tagging — normal vs. promotional

Join the date spine to transactions and promotions. Each day is tagged as either "normal" (no active promotion) or "pro" (promotion active). The ASP fallback chain is important: if no transaction ASP exists, use the promotion price; if no promotion price, use the retail price. Days before a product's first sale date are excluded to avoid polluting averages with structural zero-sales periods.

BigQuery SQL
_prep AS (
  SELECT DISTINCT
    barcode,
    doc_date,
    fsd,
    -- Most recent date this barcode appeared in each promotion theme
    MAX(CASE WHEN promotion_id IS NOT NULL THEN doc_date END)
      OVER (PARTITION BY barcode, promotion_theme)           AS last_pro,
    -- Tag: is this day a normal day or a promotion day?
    CASE WHEN promotion_id IS NULL THEN "normal" ELSE "pro" END  AS pro,
    CASE WHEN promotion_id IS NULL THEN "normal" ELSE promotion_theme END AS pro_name,
    IFNULL(sales, 0)                                           AS sales,
    -- ASP fallback: transaction ASP → promotion price → retail price
    IFNULL(IFNULL(asp, promotion_price), retail_price)         AS asp

  FROM       _id
  LEFT JOIN _receipt USING (barcode, doc_date)
  LEFT JOIN _pro     USING (barcode, doc_date)
  LEFT JOIN _product USING (barcode)

  -- Only include days from the SKU's first sale onward
  WHERE doc_date >= fsd
),
06 – 07 Average by period type & self-join

Average daily sales and ASP are computed per barcode × period type. The self-join then aligns each SKU's normal-period averages against each of its promotion-theme averages in a single row. The join condition handles the EDLP edge case: if a SKU has only ever been on promotion (no normal period exists, xx = 1), the join uses a same-theme match instead of a normal vs. promo split.

BigQuery SQL
_a AS (
  SELECT DISTINCT
    barcode,
    pro,
    pro_name                                                  AS promotion_theme,
    last_pro,
    COUNT(DISTINCT doc_date)                                  AS days,
    AVG(sales)                                                AS sales,
    AVG(asp)                                                  AS asp,
    -- xx = 1 → SKU has only promo periods (EDLP / always-on)
    -- xx = 2 → SKU has both normal and promo periods (standard case)
    COUNT(DISTINCT pro) OVER (PARTITION BY barcode)          AS xx
  FROM _prep
  GROUP BY ALL
),

_final AS (
  SELECT DISTINCT
    a.barcode,
    b.promotion_theme,
    b.last_pro,
    a.sales    AS normal_sales,
    b.sales    AS pro_sales,
    a.asp      AS normal_asp,
    b.asp      AS pro_asp,
    a.days     AS normal_days,
    b.days     AS pro_days

  FROM _a a
  INNER JOIN _a b
    ON CASE
      -- Standard: join normal rows to promo rows for the same SKU
      WHEN a.xx = 2 THEN a.barcode = b.barcode
      -- EDLP: no normal period — compare promo theme to itself
      WHEN a.xx = 1 THEN a.barcode = b.barcode AND a.promotion_theme = b.promotion_theme
    END

  WHERE
    CASE
      WHEN a.xx = 2 THEN (a.pro = "normal" AND b.pro = "pro")
      WHEN a.xx = 1 THEN TRUE
    END
),
08 Final output — % lift & price elasticity

Compute the three key metrics. SAFE_DIVIDE is used throughout to handle zero denominators gracefully. The price elasticity sign is inverted (multiplied by −1) so that positive values indicate elastic products where price cuts drive volume — the more intuitive direction for commercial interpretation.

BigQuery SQL
_result AS (
  SELECT
    *,
    -- % change in avg. daily sales: positive = promo drove volume
    SAFE_DIVIDE(pro_sales, normal_sales) - 1                  AS pct_sales,
    -- % change in ASP: typically negative (price went down on promo)
    SAFE_DIVIDE(pro_asp, normal_asp) - 1                      AS pct_asp,
    -- Price elasticity: −(% sales change / % price change)
    -- Positive = elastic (volume responds to price). Negative = inelastic.
    SAFE_DIVIDE(
      SAFE_DIVIDE(pro_sales, normal_sales) - 1,
      SAFE_DIVIDE(pro_asp,   normal_asp)   - 1
    ) * (-1)                                                  AS price_elasticity

  FROM _final
)

-- Final select: drop internal join flags, order by SKU
SELECT * EXCEPT (pro_a, pro_b)
FROM   _result
ORDER BY barcode
