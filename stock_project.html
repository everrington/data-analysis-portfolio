Most dashboards start at the visualization layer. This project owns the entire data journey — from raw inventory records sitting in a cloud data warehouse, all the way to a live, auto-refreshing Power BI report consumed daily by commercial teams.

I designed and implemented a fully automated stock out-of-stock (OOS) monitoring pipeline for a retail FMCG chain. SQL — Data Pipeline
01Fact table — incremental stock update

The fact table stores daily balance (stock quantity + value) per branch × SKU. Running a full refresh every day would be expensive. Instead, this query implements an incremental update strategy: it builds a temp table for the latest day only, deletes any existing records for that date, appends the new data, then purges anything older than 1 month — keeping the table lean and the rolling window clean. A PARTITION BY date + CLUSTER BY branch ensures BigQuery reads only the relevant date partitions.

BigQuery SQL — Scheduled Daily
-- ─────────────────────────────────────────────────────────────
-- FACT TABLE: Daily stock balance — incremental update
-- Strategy: build temp → delete existing → append → purge old
-- Runs automatically via Google Cloud Scheduler (daily)
-- ─────────────────────────────────────────────────────────────

-- Step 1: Build temp table for today's data only
CREATE OR REPLACE TABLE `[project].[dataset].fact_balances_update`
PARTITION BY  doc_date
CLUSTER BY    branch_code
AS (
  WITH

  params AS (
    SELECT
      DATE_SUB(CURRENT_DATE("Asia/Bangkok"), INTERVAL 1 DAY) AS st_dt,
      CURRENT_DATE("Asia/Bangkok")                             AS en_dt
  ),

  -- Filter to only active, registered SKUs
  _product AS (
    SELECT DISTINCT material_code
    FROM  `[project].[dataset].dim_product`
  )

  SELECT DISTINCT
    a.date           AS doc_date,
    a.location       AS branch_code,
    a.product        AS material_code,
    a.balance        AS stock_qty,
    a.value          AS stock_value

  FROM  `[source_project].[source_dataset].balances` a
  INNER JOIN _product c ON c.material_code = a.product

  WHERE
    a.date >= (SELECT st_dt FROM params)
    AND a.date <  (SELECT en_dt FROM params)
);

-- Step 2: Delete existing records for the same date (avoid duplicates)
DELETE FROM `[project].[dataset].fact_balances`
WHERE doc_date IN (
  SELECT DISTINCT doc_date
  FROM `[project].[dataset].fact_balances_update`
);

-- Step 3: Purge records older than 1 month (keep window lean)
DELETE FROM `[project].[dataset].fact_balances`
WHERE doc_date <= DATE_SUB(CURRENT_DATE("Asia/Bangkok"), INTERVAL 1 MONTH);

-- Step 4: Append today's fresh data
INSERT INTO `[project].[dataset].fact_balances`
SELECT * FROM `[project].[dataset].fact_balances_update`;

-- Step 5: Drop temp table
DROP TABLE IF EXISTS `[project].[dataset].fact_balances_update`;
02Dimension table — product master (scheduled full refresh)

The product dimension is a full refresh — it rebuilds from scratch on each scheduled run because product attributes (status, tier, brand, vendor) change frequently and slowly enough that incremental tracking adds more complexity than value. The query joins four sources to assemble a single enriched product record: the core barcode master, a supplementary Google Sheet for attributes not yet in the system (packtype, product group), a tier classification view, and a first-sale-date lookup. OEM flag is derived from vendor name pattern matching.

BigQuery SQL — Scheduled Full Refresh
-- ─────────────────────────────────────────────────────────────
-- DIMENSION TABLE: Product master — scheduled full refresh
-- Joins 4 sources: barcode master, enrichment sheet,
--                  tier classification, first sales date
-- ─────────────────────────────────────────────────────────────

CREATE OR REPLACE TABLE `[project].[dataset].dim_product`
AS
SELECT
  a.barcode,
  a.material_code,
  a.material_name,
  a.product_name,

  -- Category hierarchy
  a.subcategory_code,
  a.subcategory_name,
  a.category_code,
  a.category_name,
  a.division_id,
  a.division_name,

  -- Enriched attributes from supplementary source
  IFNULL(UPPER(b.product_group),  "no_data") AS product_group,
  IFNULL(UPPER(b.packtype_group), "no_data") AS packtype_group,
  b.product_family,

  -- Brand (prefer system value, fall back to enrichment source)
  IFNULL(UPPER(a.brand_name), UPPER(b.brand_name)) AS brand_name,
  a.brand_code,

  -- Vendor (3-level coalesce: system → purchase → enrichment)
  IFNULL(a.vendor_code, b.vendor_pur_code)                       AS vendor_code,
  IFNULL(IFNULL(a.vendor_name, b.vendor_pur_name), b.vendor_name) AS vendor_name,

  -- OEM flag: derived from vendor name pattern (not stored in system)
  CASE
    WHEN a.vendor_name   LIKE '%[OEM Vendor Pattern]%' THEN TRUE
    WHEN b.vendor_name   LIKE '%[OEM Vendor Pattern]%' THEN TRUE
    ELSE FALSE
  END AS oem,

  a.product_status,
  a.retail_price,
  a.unit_code,

  -- Tier from latest tier classification view
  COALESCE(c.tier_product, "no_tier") AS tier_product,

  -- First sales date from separate lookup table
  d.first_sales_date

FROM       `[source].barcode_master`                 a
LEFT JOIN `[project].[dataset].product_enrichment`  b ON a.material_code = b.material_code
LEFT JOIN `[project].[dataset].view_tier_latest`    c ON a.material_code = c.material_code
LEFT JOIN `[project].[dataset].sku_first_sales_date` d ON a.material_code = d.material_code

WHERE
  division_id LIKE "[Target Division]"
  AND a.product_status IS NOT NULL;
BigQuery → Power BI via MS Fabric

Once data is clean in BigQuery, Microsoft Fabric connects to BigQuery using the native GoogleBigQuery API connector — no file exports, no manual data pulls. The semantic model in Fabric imports the tables on a daily schedule, runs the DAX measure layer on top, and serves the Power BI report. The full pipeline from raw source to live report is zero-touch after setup.
