The Shop Unlimited campaign rewarded the highest-spending customers over a 31-day window. The challenge wasn't tracking spend — it was distinguishing customers who were genuinely racing toward the reward threshold from those who simply happened to transact during the period.

To solve this, I built a multi-signal behavioral intent scoring model (v7) in BigQuery. The model compares each customer's campaign-period behavior against their own 60-day pre-campaign baseline — scoring them across four independent dimensions: spend acceleration, visit acceleration, spend velocity trend, and basket size uplift. A bonus is applied for customers showing a late-stage surge in Week 4, a strong signal of intentional behavior.

Customers who clear all gate thresholds and fire at least 3 signals are classified as Intentional Racers — the highest-priority segment for commercial intervention. The output feeds directly into the Power BI dashboard for real-time monitoring and stakeholder review.

Scoring SQL · v7

The query is structured as a chain of CTEs, each building on the last. Below is each section with an explanation of the logic.

01 Pre-campaign baseline

Aggregate each customer's transaction history over the 60-day period before the campaign. Spend and trips are then normalized to a 30-day equivalent so they can be directly compared against the 30-day campaign window — eliminating the time-length bias that would otherwise inflate campaign numbers.

BigQuery SQL 
-- Baseline: [DATE_RANGE_1] — 60 days
-- Test window: [DATE_RANGE_2] — 30 days

WITH pre_campaign AS (
  SELECT
    customer_code,
    COUNT(DISTINCT doc_no)                                             AS pre_trips,
    ROUND(SUM(sales_exc_vat), 0)                                       AS pre_spend,
    ROUND(SUM(sales_exc_vat) / NULLIF(COUNT(DISTINCT doc_no), 0), 0)  AS pre_basket,

    -- Normalize 60-day spend → 30-day equivalent
    ROUND(SUM(sales_exc_vat) / 60.0 * 30, 0)                          AS pre_spend_30d_equiv,
    ROUND(COUNT(DISTINCT doc_no) / 60.0 * 30, 1)                       AS pre_trips_30d_equiv

  FROM `[project].[dataset].transactions`
  WHERE
    doc_date BETWEEN '[BASELINE_START]' AND '[BASELINE_END]'
    AND customer_code IS NOT NULL
    AND customer_code != '0'
  GROUP BY 1
),
02 Campaign weekly buckets

Split the 30-day campaign window into 4 weekly buckets (W1–W4). This granularity is critical — it's what allows the model to detect late-stage behavioral surges, which are the strongest signal of intentional campaign participation.

BigQuery SQL 
campaign_weekly AS (
  SELECT
    customer_code,
    CASE
      WHEN doc_date BETWEEN '[W1_START]' AND '[W1_END]' THEN 'W1'
      WHEN doc_date BETWEEN '[W2_START]' AND '[W2_END]' THEN 'W2'
      WHEN doc_date BETWEEN '[W3_START]' AND '[W3_END]' THEN 'W3'
      WHEN doc_date BETWEEN '[W4_START]' AND '[W4_END]' THEN 'W4'
    END AS camp_week,
    SUM(sales_exc_vat)     AS weekly_spend,
    COUNT(DISTINCT doc_no) AS weekly_trips

  FROM `[project].[dataset].transactions`
  WHERE
    doc_date BETWEEN '[CAMPAIGN_START]' AND '[CAMPAIGN_END]'
    AND customer_code IS NOT NULL
    AND customer_code != '0'
  GROUP BY 1, 2
),

campaign_total AS (
  SELECT
    customer_code,
    SUM(weekly_spend)  AS camp_spend,
    SUM(weekly_trips)  AS camp_trips,

    -- W3+W4 vs W1+W2: measures whether spend is accelerating across the campaign
    ROUND(
      SUM(CASE WHEN camp_week IN ('W3','W4') THEN weekly_spend ELSE 0 END) /
      NULLIF(SUM(CASE WHEN camp_week IN ('W1','W2') THEN weekly_spend ELSE 0 END), 0)
    , 2) AS spend_velocity_ratio,

    -- W4 vs W1: detects end-of-campaign sprint (late-surge signal)
    ROUND(
      SUM(CASE WHEN camp_week = 'W4' THEN weekly_spend ELSE 0 END) /
      NULLIF(SUM(CASE WHEN camp_week = 'W1' THEN weekly_spend ELSE 0 END), 0)
    , 2) AS w4_vs_w1_ratio

  FROM campaign_weekly
  GROUP BY 1, 2
),
03 Multi-signal scoring model

Each customer is scored across four independent dimensions, each benchmarked against their own pre-campaign baseline rather than a population average. This makes the model self-calibrating — a customer who triples their normal spend scores identically regardless of their absolute spend level. Gate flags are also computed here as hard eligibility checks applied later.

BigQuery SQL 
scored AS (
  SELECT
    ct.customer_code,
    ct.camp_spend,    ct.camp_trips,
    ct.spend_velocity_ratio, ct.w4_vs_w1_ratio,
    p.pre_spend,      p.pre_trips,
    p.pre_spend_30d_equiv, p.pre_trips_30d_equiv,

    -- [1] SPEND ACCELERATION (max 30 pts)
    -- How much did campaign spend exceed normalized baseline spend?
    CASE
      WHEN p.pre_spend_30d_equiv IS NULL OR p.pre_spend_30d_equiv < 200 THEN 0
      WHEN ct.camp_spend >= p.pre_spend_30d_equiv * 5.0  THEN 30
      WHEN ct.camp_spend >= p.pre_spend_30d_equiv * 4.0  THEN 22
      WHEN ct.camp_spend >= p.pre_spend_30d_equiv * 3.0  THEN 14
      WHEN ct.camp_spend >= p.pre_spend_30d_equiv * 2.0  THEN 7
      WHEN ct.camp_spend >= p.pre_spend_30d_equiv * 1.5  THEN 3
      ELSE 0
    END AS score_spend_accel,

    -- [2] VISIT ACCELERATION (max 25 pts)
    -- Did the customer visit significantly more often during the campaign?
    CASE
      WHEN p.pre_trips_30d_equiv IS NULL OR p.pre_trips_30d_equiv < 2 THEN 0
      WHEN ct.camp_trips >= p.pre_trips_30d_equiv * 4.0  THEN 25
      WHEN ct.camp_trips >= p.pre_trips_30d_equiv * 3.0  THEN 18
      WHEN ct.camp_trips >= p.pre_trips_30d_equiv * 2.0  THEN 11
      WHEN ct.camp_trips >= p.pre_trips_30d_equiv * 1.5  THEN 5
      WHEN ct.camp_trips >= p.pre_trips_30d_equiv * 1.2  THEN 2
      ELSE 0
    END AS score_visit_accel,

    -- [3] SPEND VELOCITY (max 25 pts)
    -- Is spend accelerating across the campaign weeks (W3+W4 vs W1+W2)?
    CASE
      WHEN ct.spend_velocity_ratio IS NULL    THEN 0
      WHEN ct.spend_velocity_ratio >= 5.0     THEN 25
      WHEN ct.spend_velocity_ratio >= 4.0     THEN 18
      WHEN ct.spend_velocity_ratio >= 3.0     THEN 12
      WHEN ct.spend_velocity_ratio >= 2.5     THEN 7
      WHEN ct.spend_velocity_ratio >= 2.0     THEN 3
      ELSE 0
    END AS score_velocity,

    -- [4] BASKET SIZE UPLIFT (max 20 pts)
    -- Did spend-per-visit increase vs baseline? Signals consolidation behavior.
    CASE
      WHEN p.pre_trips_30d_equiv IS NULL OR p.pre_trips_30d_equiv < 2 THEN 0
      WHEN (ct.camp_spend / NULLIF(ct.camp_trips, 0))
           >= (p.pre_spend_30d_equiv / NULLIF(p.pre_trips_30d_equiv, 0)) * 3.0  THEN 20
      WHEN (ct.camp_spend / NULLIF(ct.camp_trips, 0))
           >= (p.pre_spend_30d_equiv / NULLIF(p.pre_trips_30d_equiv, 0)) * 2.5  THEN 14
      WHEN (ct.camp_spend / NULLIF(ct.camp_trips, 0))
           >= (p.pre_spend_30d_equiv / NULLIF(p.pre_trips_30d_equiv, 0)) * 2.0  THEN 8
      WHEN (ct.camp_spend / NULLIF(ct.camp_trips, 0))
           >= (p.pre_spend_30d_equiv / NULLIF(p.pre_trips_30d_equiv, 0)) * 1.5  THEN 3
      ELSE 0
    END AS score_basket_uplift,

    -- [BONUS] END-OF-CAMPAIGN SURGE (+5 pts)
    -- W4 spend > 4× W1 spend: strong late-stage sprint signal
    CASE WHEN ct.w4_vs_w1_ratio > 4.0 THEN 5 ELSE 0 END AS bonus_end_surge,

    -- GATE FLAGS (hard eligibility checks — all must pass for top tiers)
    CASE WHEN ct.camp_spend  >= 1500 THEN 1 ELSE 0 END AS gate_min_spend,
    CASE WHEN ct.camp_trips  >= 4    THEN 1 ELSE 0 END AS gate_min_trips,
    CASE WHEN p.pre_spend    >= 400  THEN 1 ELSE 0 END AS gate_min_baseline,
    CASE WHEN p.pre_trips    >= 4    THEN 1 ELSE 0 END AS gate_baseline_trips

  FROM campaign_total ct
  LEFT JOIN pre_campaign p USING (customer_code)
),
04 Signal breadth & tier classification

Signals fired counts how many dimensions scored above a meaningful threshold — this prevents a single extreme dimension from inflating the tier. The final classification requires both a score threshold and a minimum number of signals fired, ensuring tier assignment is robust and multi-dimensional.

BigQuery SQL 
signal_breadth AS (
  SELECT
    *,
    -- Total intent score (max 105 pts including bonus)
    (score_spend_accel + score_visit_accel + score_velocity
     + score_basket_uplift + bonus_end_surge) AS intent_score,

    -- Count of dimensions that fired above meaningful threshold
    (CASE WHEN score_spend_accel   >= 18 THEN 1 ELSE 0 END +
     CASE WHEN score_visit_accel   >= 15 THEN 1 ELSE 0 END +
     CASE WHEN score_velocity      >= 15 THEN 1 ELSE 0 END +
     CASE WHEN score_basket_uplift >= 12 THEN 1 ELSE 0 END)
      AS signals_fired

  FROM scored
)

-- FINAL OUTPUT: tier summary with customer count and avg metrics
SELECT
  intent_tier,
  COUNT(*)                                                             AS customer_count,
  ROUND(COUNT(*) * 100.0 / SUM(COUNT(*)) OVER(), 2)               AS pct_of_total,
  ROUND(AVG(camp_spend), 0)                                           AS avg_camp_spend,
  ROUND(AVG(camp_trips), 1)                                           AS avg_camp_trips,
  ROUND(AVG(intent_score), 1)                                         AS avg_intent_score
FROM (
  SELECT
    *,
    CASE
      WHEN intent_score     >= 80
       AND gate_min_spend    = 1
       AND gate_min_trips    = 1
       AND gate_min_baseline = 1
       AND gate_baseline_trips = 1
       AND signals_fired     >= 3  THEN 'Intentional racer'
      WHEN intent_score     >= 55
       AND gate_min_baseline = 1
       AND gate_baseline_trips = 1
       AND signals_fired     >= 3  THEN 'Active climber'
      WHEN intent_score     >= 25
       AND gate_min_baseline = 1     THEN 'Passive beneficiary'
      ELSE                              'Accidental qualifier'
    END AS intent_tier
  FROM signal_breadth
) t
GROUP BY 1
ORDER BY
  CASE intent_tier
    WHEN 'Intentional racer'   THEN 1
    WHEN 'Active climber'      THEN 2
    WHEN 'Passive beneficiary' THEN 3
    ELSE 4
  END
