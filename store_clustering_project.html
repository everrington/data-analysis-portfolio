Store Clustering to Unlock
Premium Shampoo Opportunities

BigQuery SQL Retail Strategy Customer Analytics

Not every store is equally ready for premium products. Placing high-priced SKUs in the wrong store wastes shelf space and promotional budget — while missing the right stores means leaving revenue on the table. The challenge is that "readiness" isn't a single number: it's a combination of who is shopping there, how much they spend per visit, and how productively the category is performing per shelf bay.

This project builds a composite store ranking model in BigQuery that scores every branch across three independently derived signals — premium customer penetration, category sales per bay, and average basket size — then combines them into a single weighted score using normalized percentages. The result is a ranked list of stores prioritized for premium shampoo range expansion, planogram upgrades, or targeted promotional investment.

SQL walkthrough

The query is one BigQuery SQL script structured as a chain of CTEs — each building a clean intermediate result that the next step consumes. Below each section is explained before the code.

01Variables, product filter & branch filter

Define the 6-month analysis window in a single place so dates never need to be hardcoded elsewhere. Product filter targets active shampoo and conditioner SKUs only — discontinued statuses are excluded. A key technique here: ml is extracted from the material name using regex (REGEXP_EXTRACT) because pack size isn't stored as a separate column in the product master, but is always embedded in the name string (e.g. "Brand X Shampoo 400ml"). Branch filter excludes specific store formats and tags each store as new or old based on opening date.

BigQuery SQL
WITH

-- ── Date window ───────────────────────────────────────────
_var AS (
  SELECT
    DATE('[START_DATE]') AS std,
    DATE('[END_DATE]')   AS endd
),

-- ── Active shampoo + conditioner SKUs ─────────────────────
-- ml extracted via regex — pack size lives in the name string
_product AS (
  SELECT DISTINCT
    barcode,
    material_code,
    material_name,
    brand_name,
    category_name,
    SAFE_CAST(
      REGEXP_EXTRACT(material_name, r'([0-9]+)ml')
    AS INT64) AS ml     -- e.g. "Brand X 400ml" → 400

  FROM `[project].[dataset].product_master`
  WHERE
    category_name IN ('SHAMPOO', 'CONDITIONER')
    AND material_name NOT LIKE '%+%'          -- exclude bundle packs
    AND product_status NOT IN ('C','CP','CD','D')  -- exclude discontinued
),

-- ── Shampoo+conditioner barcodes only (for basket calc) ───
_shamp AS (
  SELECT DISTINCT barcode
  FROM `[project].[dataset].product_master`
  WHERE
    category_name IN ('SHAMPOO', 'CONDITIONER')
    AND material_name NOT LIKE '%+%'
),

-- ── Branch master — tag new vs. existing stores ───────────
_branch AS (
  SELECT
    *,
    CASE WHEN opening_date >= '[NEW_STORE_CUTOFF]'
         THEN 'new' ELSE 'old'
    END AS new_store
  FROM `[project].[dataset].branch_master`
  WHERE storetype NOT IN ('[Excluded Format]')
),
02POG assortment — filter to urban stores, latest snapshot

The POG (planogram) master records which SKUs are assigned to which stores, and how many bays each store has allocated to the category. Two important filters are applied here: only the latest month's snapshot is used (pog_on_month), and standalone display units ("Stand") are excluded — those are temporary fixtures, not the main shelf, so including them would overcount bay space and distort the sales-per-bay metric.

BigQuery SQL
_pog AS (
  SELECT DISTINCT
    pog_category,
    pog_name,
    total_bay,
    barcode,
    CASE
      WHEN pog_name LIKE '%Stand%'  THEN 'stand'
      WHEN pog_name LIKE '%[Urban Keyword]%' THEN 'urban'
      ELSE 'other'
    END AS store_type,
    branch_code

  FROM `[project].[dataset].pog_master`
  WHERE
    pog_category  = '[Target Category]'
    AND pog_on_month = '[LATEST_MONTH]'

  GROUP BY ALL
  HAVING store_type <> 'stand'   -- exclude temporary display fixtures
),

-- ── Transaction base table ────────────────────────────────
_receipt AS (
  SELECT
    doc_date, doc_no, customer_code,
    barcode, branch_code,
    sales_exc_vat,
    sales_exc_vat + sales_vat AS sales,
    sales_qty
  FROM `[project].[dataset].transactions`
  WHERE
    doc_date BETWEEN (SELECT std FROM _var) AND (SELECT endd FROM _var)
    AND sales_qty     <> 0
    AND sales_exc_vat <> 0
),
03Signal 1 — Premium customer penetration

This is the most analytically interesting signal. "Premium" is not defined by price tier alone — it's derived from each customer's spend-per-ml ratio across their category purchases. A customer who buys ฿200 for 400ml is paying ฿0.5/ml, exactly at the premium threshold. A customer who buys ฿80 for 400ml (฿0.2/ml) is buying mass-market. By computing this ratio per customer, the model identifies stores where shoppers are already demonstrating premium buying behavior — the strongest signal that premium range expansion will perform.

BigQuery SQL
-- Step 1: compute spend-per-ml ratio per customer per branch
_premium_raw AS (
  SELECT DISTINCT
    g.branch_code,
    customer_code,
    SUM(sales) / SUM(ml * sales_qty) AS spend_per_ml
                -- e.g. 200 / (400 × 1) = 0.50 ฿/ml → premium
  FROM       _receipt  r
  INNER JOIN _product  p ON r.barcode      = p.barcode
  INNER JOIN _pog      g ON r.branch_code  = g.branch_code
  GROUP BY ALL
  HAVING spend_per_ml IS NOT NULL
),

-- Step 2: classify each customer as premium or normal
_premium_classified AS (
  SELECT DISTINCT
    branch_code,
    CASE WHEN spend_per_ml >= 0.5 THEN 'premium' ELSE 'normal' END AS segment,
    COUNT(DISTINCT customer_code) AS customer_count
  FROM _premium_raw
  GROUP BY ALL
),

-- Step 3: compute % premium per branch
_premium_pct AS (
  SELECT DISTINCT
    branch_code,
    customer_count / SUM(customer_count) OVER (PARTITION BY branch_code) AS pct_premium
  FROM _premium_classified
  WHERE segment = 'premium'   -- keep only the premium row per branch
),
04Signal 2 — Category sales per shelf bay

Sales/bay measures how productively each store converts its allocated shelf space into revenue. The join here requires both the branch code AND barcode to match between the transaction and the POG — this ensures only sales of POG-assorted products are counted, preventing a store with high total sales (perhaps from promotions) from being credited with productivity on a shelf that isn't performing.

BigQuery SQL
_sales_per_bay AS (
  SELECT DISTINCT
    r.branch_code,
    total_bay,
    pog_name,
    SUM(sales_exc_vat) / total_bay AS sales_per_bay
                -- double join: branch AND barcode must match POG
  FROM       _receipt  r
  INNER JOIN _pog      g ON r.branch_code = g.branch_code
                        AND r.barcode     = g.barcode
  GROUP BY ALL
),
05Signal 3 — Average basket size

Average basket size is computed at the customer level first (total category spend ÷ distinct transactions per customer), then averaged across all customers in the branch. Computing it this way — customer-average rather than a raw per-store average — prevents a small number of very high-spend customers from inflating the metric and misrepresenting the typical shopper experience in that store.

BigQuery SQL
_avg_basket AS (
  SELECT DISTINCT
    branch_code,
    AVG(basket_per_customer) AS avg_basket
  FROM (
    -- inner: basket size per customer per branch
    SELECT
      r.branch_code,
      r.customer_code,
      SUM(sales) / COUNT(DISTINCT doc_no) AS basket_per_customer

    FROM       _receipt  r
    INNER JOIN _branch   b ON r.branch_code = b.branch_code
    INNER JOIN _shamp    s ON r.barcode     = s.barcode
    GROUP BY ALL
  )
  GROUP BY ALL
),
06Final scoring — normalize, weight & rank

The three signals are on completely different scales — a percentage, a sales figure, and a basket size. Combining them directly would let the largest numbers dominate. The solution: normalize each signal by dividing by its own SUM() OVER(), which converts every value to a proportion of the total across all branches. Each store's normalized score now represents its fair share of that metric. The weighted sum is then ranked using DENSE_RANK() so tied stores share the same rank position.

BigQuery SQL
-- Assemble all three signals per branch, normalize, then score
_scored AS (
  SELECT DISTINCT
    b.branch_code,
    b.new_store,
    s.total_bay,
    c.pct_premium                                           AS premium_cust_pct,
    s.sales_per_bay,
    z.avg_basket,

    -- Normalize each signal to its share of the total across all branches
    c.pct_premium  / SUM(c.pct_premium)  OVER()             AS norm_premium,
    s.sales_per_bay / SUM(s.sales_per_bay) OVER()            AS norm_sales,
    z.avg_basket    / SUM(z.avg_basket)    OVER()            AS norm_basket

  FROM       _branch        b
  INNER JOIN _premium_pct   c ON b.branch_code = c.branch_code
  INNER JOIN _sales_per_bay s ON b.branch_code = s.branch_code
  INNER JOIN _avg_basket    z ON b.branch_code = z.branch_code
)

-- Final output: composite score + rank
SELECT
  branch_code,
  new_store,
  total_bay,
  premium_cust_pct,
  sales_per_bay,
  avg_basket,

  -- Weighted composite score (signals already normalized)
  norm_premium * 0.4
  + norm_sales  * 0.3
  + norm_basket * 0.3                                        AS composite_score,

  DENSE_RANK() OVER (
    ORDER BY
      norm_premium * 0.4
      + norm_sales  * 0.3
      + norm_basket * 0.3 DESC
  )                                                           AS store_rank

FROM _scored
ORDER BY new_store, store_rank
